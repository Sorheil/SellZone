1. INTRODUCTION

1.1 But du document

Ce document sert de guide complet de développement pour l'application SellZone. Il définit l'ensemble des spécifications fonctionnelles et techniques nécessaires à la réalisation du logiciel, en suivant une approche méthodologique structurée. Il constitue la référence commune entre le développeur et les parties prenantes.

1.2 Contexte de l'application

1.2.1 Problématique et objectifs

L'application SellZone est conçue pour servir de template/modèle standardisé de boutique e-commerce. Elle répond aux besoins croissants des petits commerçants qui souhaitent développer leur présence commerciale en ligne, mais qui manquent d'une plateforme simple et intégrée pour :

  • Créer et gérer leur boutique en ligne de manière autonome
  • Vendre leurs produits ou services facilement
  • Gérer les commandes et les clients
  • Accroître leur chiffre d'affaires

1.2.2 Modèle économique

SellZone est avant tout un service de création de sites e-commerce. Le modèle économique repose sur :

  • Service principal : Création et personnalisation d'une boutique en ligne basée sur ce template
  • Processus : À partir du template SellZone, chaque client (vendeur) reçoit une instance personnalisée du site
  • Monétisation : Chaque nouvelle boutique créée génère des revenus pour le développeur/prestataire

1.2.3 Portée et environnement

  • Périmètre géographique : Contexte national (camerounais)
  • Utilisateurs cibles (Vendeurs) : Petits commerçants de tous types (vêtements, épiceries, décoration, services, artisanat, etc.)
  • Utilisateurs finaux (Acheteurs) : Clients en ligne souhaitant acheter les produits/services du vendeur
  • Types de produits/services : Tous types de produits et services commercialisables (sauf variantes)
  • Volume de catalogue : Catalogues simples de moins de 100 produits en moyenne

1.2.4 Phase actuelle

Ce document décrit la version 1.0 du template SellZone. À cette phase :

  • Il s'agit d'une boutique e-commerce mono-vendeur (un seul commerçant par instance)
  • La plateforme elle-même n'est pas multi-clients à ce stade ; elle servira de base/modèle
  • Les évolutions futures (scalabilité, multi-clients, API, etc.) sont anticipées mais hors scope

2. SPÉCIFICATIONS GÉNÉRALES

2.1 Présentation du logiciel

2.1.1 Environnement du logiciel

Architecture technique globale :

  • Backend : Spring Boot (Java)
  • Frontend : Angular (TypeScript)
  • Base de données : PostgreSQL
  • Cache : Redis (pour sessions et optimisations futures)
  • Infrastructure : AWS (Cloud)

Contexte du système :

Entité externe Interaction avec le système
Acheteur (Client) Navigation, recherche produits, panier, paiement, suivi commandes
Vendeur (Admin boutique) Gestion produits, suivi commandes, exports, configuration via dashboard
Agrégateur de paiement Redirection pour Mobile Money, confirmation de paiement via webhook
Infrastructure AWS Hébergement, base de données, cache, sécurité

2.1.2 Description générale du logiciel

SellZone est une application web complète de e-commerce permettant :

Pour l'acheteur (Frontend) :

  • Consulter le catalogue de produits avec description, prix, images
  • Rechercher et filtrer les produits
  • Ajouter/retirer des produits au panier (panier persistant)
  • Voir la disponibilité des produits en temps réel
  • Créer un compte utilisateur et se connecter
  • Passer commande en ligne
  • Effectuer le paiement (Mobile Money ou paiement à la livraison)
  • Télécharger la facture électronique
  • Consulter l'historique de ses commandes
  • Laisser des avis et commentaires sur les produits

Pour le vendeur (Backend/Dashboard) :

  • Créer, modifier, supprimer des produits
  • Gérer les images et descriptions des produits
  • Gérer les stocks (voir, mettre à jour)
  • Consulter un dashboard avec statistiques (nombre de commandes, chiffre d'affaires)
  • Consulter et gérer les commandes (changer les statuts)
  • Consulter l'historique des paiements
  • Exporter les données (commandes, clients, etc.)
  • Télécharger les factures

2.2 Contraintes opérationnelles

2.2.1 Performances

  • pour le moment aucune Contraintes de Performances n'est d'actualité car pas encore repertori

2.2.2 Installation et déploiement

  • Environnement : Déploiement cloud sur AWS (conteneurisation Docker envisagée)
  • Base de données : Migration SQL automatisée à chaque déploiement

2.3 Contraintes de réalisation

2.3.1 Contraintes structurelles

  • Architecture : Séparation claire Backend/Frontend (REST API)
  • Base de données : PostgreSQL avec migrations versionnées
  • ORM : JPA/Hibernate pour la couche données
  • Stateless : Backend stateless, sessions gérées via JWT + cookies

2.3.2 Contraintes d'évolution

  • Le logiciel est conçu comme un template réutilisable, facilement adaptable pour de nouveaux clients
  • Possibilité de migration future vers une architecture multi-clients (marketplace)
  • Versioning des API REST dès le départ (v1, v2, etc.)
  • Structure de code modulaire pour faciliter maintenabilité et extensions

2.3.3 Contraintes de développement

  • Développeur : Actuellement seul développeur, mais structure anticipant du travail en équipe
  • Délai : MVP à livrer en 1-2 semaines
  • Stack : Spring Boot, Angular, PostgreSQL, AWS, Redis
  • Méthodologie : Approche itérative, focus sur MVP fonctionnel
  • Versioning : Git/GitHub pour le contrôle de source

2.3.4 Contraintes de qualité

  • Sécurité : Authentification JWT, chiffrement des données sensibles (mots de passe, paiements)
  • Fiabilité : Gestion des erreurs robuste, messages clairs aux utilisateurs
  • Testabilité : Tests unitaires et d'intégration pour les modules critiques (panier, paiement)
  • Maintenabilité : Code bien structuré, commenté, followant les conventions
  • Accessibilité : Interface responsive, compatibilité navigateurs modernes

3.4 Modules fonctionnels détaillés

Module 1 : Authentification et Gestion Utilisateurs

1.1 Fonction : Inscription Acheteur

Présentation : Permet à un nouvel acheteur de créer un compte pour accéder à toutes les fonctionnalités de la boutique.

Entrées :

  • Email (unique)
  • Nom
  • Prénom
  • Mot de passe (minimum 8 caractères)
  • Confirmation mot de passe

Traitements :

  • Validation des données (format email, longueur mot de passe, correspondance des mots de passe)
  • Vérification unicité de l'email
  • Hachage du mot de passe (bcrypt ou similaire)
  • Création de l'entité utilisateur en base
  • Génération JWT pour session automatique

Sorties :

  • Compte créé avec ID unique
  • JWT retourné pour connexion automatique
  • Message de confirmation

1.2 Fonction : Connexion

Présentation : Authentifie un utilisateur (acheteur ou vendeur) via email et mot de passe.

Entrées :

  • Email
  • Mot de passe

Traitements :

  • Recherche utilisateur par email
  • Vérification du mot de passe (bcrypt compare)
  • Génération JWT + cookie HTTP-only sécurisé
  • Enregistrement dernière connexion

Sorties :

  • JWT et cookie d'authentification
  • Informations utilisateur (email, rôle, etc.)
  • Statut : succès ou erreur (identifiants invalides)

1.3 Fonction : Déconnexion

Présentation : Termine la session de l'utilisateur.

Entrées : JWT/Cookie utilisateur

Traitements :

  • Invalidation du JWT
  • Suppression du cookie

Sorties : Confirmation de déconnexion

Module 2 : Gestion du Catalogue (Produits)

2.1 Fonction : Créer un produit

Présentation : Le vendeur crée un nouveau produit dans son catalogue.

Entrées :

  • Nom du produit
  • Description
  • Prix (en devise locale)
  • Quantité en stock
  • Images (1 principale, optionnelles additionnelles)
  • Catégorie (optionnel pour V1)

Traitements :

  • Validation des données obligatoires
  • Upload et stockage des images (AWS S3 ou équivalent)
  • Création de l'enregistrement produit
  • Génération d'ID unique et slug URL-friendly
  • Indexation pour recherche

Sorties :

  • Produit créé avec ID
  • URL d'accès au produit
  • Confirmation et redirection

2.2 Fonction : Modifier un produit

Présentation : Le vendeur met à jour les informations d'un produit existant.

Entrées : ID produit + champs à modifier (nom, description, prix, stock, images)

Traitements :

  • Vérification des permissions (propriétaire de la boutique)
  • Validation des nouvelles données
  • Mise à jour en base
  • Gestion des images : upload nouvelles, suppression anciennes

Sorties : Produit mis à jour, confirmation

2.3 Fonction : Supprimer un produit

Présentation : Le vendeur supprime définitivement un produit.

Entrées : ID produit

Traitements :

  • Soft delete (marquage logique) ou hard delete
  • Suppression des images associées
  • Suppression des avis associés (optionnel : archivage)

Sorties : Confirmation suppression

2.4 Fonction : Lister les produits (Acheteur)

Présentation : Affiche le catalogue de produits disponibles.

Entrées :

  • Page (pagination)
  • Terme de recherche (optionnel)
  • Filtres (prix min/max, etc.)

Traitements :

  • Requête en base avec pagination (ex. 20 produits par page)
  • Recherche texte sur nom/description si terme fourni
  • Filtrage et tri
  • Inclusion des avis et note moyenne

Sorties : Liste paginée des produits avec détails (image miniature, prix, note moyenne)

2.5 Fonction : Consulter détails produit

Présentation : Affiche la fiche complète d'un produit.

Entrées : ID ou slug produit

Traitements :

  • Récupération produit + images + avis
  • Calcul note moyenne des avis
  • Vérification disponibilité stock

Sorties : Fiche produit complète (nom, description longue, images, prix, stock, avis, bouton "Ajouter au panier" actif/inactif)

Module 3 : Panier et Commandes

3.1 Fonction : Ajouter au panier

Présentation : Ajoute un produit au panier de l'acheteur.

Entrées :

  • ID produit
  • Quantité (par défaut 1)
  • Session acheteur

Traitements :

  • Vérification stock disponible
  • Recherche produit en panier
  • Si existe : incrémenter quantité
  • Si n'existe pas : ajouter nouvelle ligne
  • Calcul total panier
  • Persistance panier (base de données ou Redis)

Sorties : Panier mis à jour, message de confirmation

3.2 Fonction : Retirer du panier

Présentation : Supprime un produit du panier.

Entrées : ID produit dans le panier

Traitements :

  • Suppression de la ligne panier
  • Recalcul total
  • Persistance

Sorties : Panier mise à jour, confirmation

3.3 Fonction : Consulter panier

Présentation : Affiche le contenu du panier actuel.

Entrées : Session acheteur

Traitements :

  • Récupération panier persisté
  • Vérification stock en temps réel pour chaque article
  • Recalcul total (prix unitaire × quantité)

Sorties : Panier avec articles, sous-total, total

3.4 Fonction : Créer une commande

Présentation : Valide le panier et crée une commande.

Entrées :

  • Panier complet
  • Mode de paiement (Mobile Money ou Cash à la livraison)
  • Informations acheteur (email, adresse de livraison si nécessaire)

Traitements :

  • Vérification finale stock pour chaque article
  • Diminution du stock en base
  • Création de la commande avec numéro unique
  • Statut initial : "En attente" (paiement non confirmé)
  • Enregistrement du montant total
  • Création de la facture électronique
  • Si Mobile Money : génération lien de paiement (appel agrégateur)
  • Vidage du panier

Sorties :

  • Commande créée avec ID unique
  • Si Mobile Money : URL de paiement (redirection)
  • Si Cash : confirmation et message "Paiement à la livraison"
  • Facture générée

3.5 Fonction : Consulter historique commandes

Présentation : L'acheteur consulte ses commandes passées.

Entrées : ID acheteur

Traitements :

  • Requête des commandes de cet acheteur
  • Tri par date décroissante
  • Inclusion du statut et du montant

Sorties : Liste des commandes avec détails (numéro, date, statut, montant, lien pour télécharger facture)

Module 4 : Paiement et Facturation

4.1 Fonction : Traiter paiement Mobile Money

Présentation : Gère les paiements via agrégateur de paiement (Mobile Money).

Entrées :

  • ID commande
  • Montant à payer
  • Numéro téléphone acheteur

Traitements :

  • Appel API agrégateur pour générer lien de paiement
  • Redirection acheteur vers agrégateur (lien externe)
  • Attente confirmation (webhook depuis agrégateur)
  • À la confirmation : mise à jour statut commande "Confirmée"
  • Enregistrement de la transaction

Sorties : Lien de paiement externe ou confirmation de paiement reçue

4.2 Fonction : Traiter paiement Cash à la livraison

Présentation : Crée une commande avec paiement différé à la livraison.

Entrées : ID commande

Traitements :

  • Commande créée avec statut "En attente"
  • Enregistrement mode de paiement "Cash"
  • Pas de redirection de paiement

Sorties : Commande enregistrée, message confirmatif au client

4.3 Fonction : Générer facture électronique

Présentation : Crée une facture PDF pour la commande.

Entrées : ID commande

Traitements :

  • Récupération données commande, articles, acheteur, vendeur
  • Génération PDF avec :
    • En-tête avec infos boutique (logo, adresse, etc.)
    • Numéro facture, date, numéro de commande
    • Informations acheteur
    • Détail des articles (nom, quantité, prix unitaire, sous-total)
    • Total TTC (TVA si applicable)
    • Mode de paiement
  • Stockage du PDF

Sorties : Fichier PDF facture téléchargeable

4.4 Fonction : Télécharger facture

Présentation : Permet à l'acheteur de télécharger la facture de sa commande.

Entrées : ID commande

Traitements :

  • Vérification permissions (propriétaire de la commande)
  • Récupération du PDF généré
  • Envoi en téléchargement (Content-Disposition: attachment)

Sorties : PDF téléchargé

Module 5 : Gestion des Commandes (Vendeur)

5.1 Fonction : Consulter les commandes

Présentation : Le vendeur voit toutes les commandes passées à sa boutique.

Entrées : Filtres optionnels (statut, date, acheteur)

Traitements :

  • Requête des commandes de cette boutique
  • Filtrage selon critères
  • Tri par date décroissante
  • Pagination

Sorties : Liste des commandes avec numéro, date, acheteur, montant, statut

5.2 Fonction : Détails commande

Présentation : Affiche le détail complet d'une commande.

Entrées : ID commande

Traitements :

  • Récupération commande + articles + info acheteur
  • Calcul totaux

Sorties : Fiche commande complète (articles, montants, adresse livraison si fournie, statut, mode de paiement)

5.3 Fonction : Changer statut commande

Présentation : Le vendeur met à jour le statut de la commande.

Entrées :

  • ID commande
  • Nouveau statut (En attente → Confirmée → Livrée)

Traitements :

  • Validation de la transition de statut
  • Mise à jour en base
  • Enregistrement de l'historique de changement
  • Notification acheteur (optionnel pour V1)

4. SPÉCIFICATIONS D'INTERFACE

4.1 Interface Homme-Machine (IHM)

4.1.1 Vue d'ensemble et principes de conception

L'application SellZone est une application web responsive accessible via navigateur standard. Les principes de conception sont :

  • Simplicité : Interface épurée, facile d'utilisation pour les commerçants non-tech
  • Responsive : Adaptée à tous les appareils (desktop, tablette, mobile)
  • Performance : Chargement rapide, animations fluides
  • Accessibilité : Contraste suffisant, textes lisibles, navigation au clavier
  • Cohérence : Même thème et navigation partout

4.1.2 Structure générale de l'interface

Pour l'acheteur :

  • Barre de navigation : Logo/nom boutique, recherche, panier (avec badge nombre articles), connexion/profil
  • Page d'accueil : Hero section (image + titre), catalogue de produits en grille, footer avec infos boutique
  • Fiche produit : Images, description, prix, stock, bouton ajouter au panier, section avis
  • Panier : Liste articles, quantités modifiables, total, bouton "Procéder"
  • Checkout : Résumé commande, choix mode de paiement, confirmation
  • Historique commandes : Tableau avec statuts, montants, liens factures

Pour le vendeur :

  • Dashboard : Cartes statistiques, graphiques, dernières commandes, raccourcis
  • Gestion produits : Tableau des produits, CRUD, bulk edit
  • Gestion commandes : Tableau des commandes, filtres, détails, changement statut
  • Barre latérale : Menu de navigation (Dashboard, Produits, Commandes, Avis, Exports, Profil)

4.1.3 Enchaînement des fenêtres et flux de navigation

Flux acheteur :

  1. Accueil → Parcourir catalogue (recherche/filtres optionnels)
  2. Fiche produit → Ajouter au panier
  3. Panier → Modifier quantités / Retirer articles
  4. Panier → Procéder à la commande
  5. Checkout → Choisir mode paiement (Mobile Money ou Cash)
  6. Si Mobile Money → Redirection agrégateur → Retour après paiement
  7. Confirmation commande → Télécharger facture
  8. Historique commandes → Consulter statuts et factures

Flux vendeur :

  1. Connexion dashboard → Dashboard principal
  2. Dashboard → Gestion Produits / Gestion Commandes / Avis
  3. Gestion Produits → CRUD produits
  4. Gestion Commandes → Détails commande → Changer statut
  5. Avis → Modérer avis inappropriés
  6. Exports → Télécharger données (CSV/Excel)

4.1.4 Contraintes d'utilisation et conditions particulières

  • Validation en temps réel : Les formulaires (inscription, création produit) affichent des messages d'erreur en temps réel
  • Confirmation destructives : Avant suppression produit ou avis, modal de confirmation
  • Panier persistant : Même après fermeture navigateur, le panier se restaure (session)
  • Stock en temps réel : Quantités disponibles mises à jour instantanément
  • Responsive : Navigation adaptée mobile (menu hamburger)
  • Loading states : Indicateurs de chargement (spinners, skeleton screens)
  • Messages utilisateur : Toast notifications pour confirmations et erreurs

4.2 Interfaces Logiciel/Logiciel

4.2.1 Architecture globale des interfaces

L'application suit une architecture REST API où le backend (Spring Boot) expose des endpoints consommés par le frontend (Angular).

Type d'interface Description

Stockage fichiers AWS S3 (ou équivalent) pour images produits et factures PDF
Cache Redis pour sessions utilisateur et optimisations

Entité : Utilisateur

User {
  id: UUID (PK)
  email: String (UNIQUE)
  password: String (hashed)
  nom: String
  prenom: String
  role: Enum (VENDEUR, ACHETEUR)
  dateCreation: Timestamp
  dateModification: Timestamp
}

Product {
  id: UUID (PK)
  vendeurId: UUID (FK → User)
  nom: String
  description: String
  prix: Decimal
  stock: Integer
  images: List[String] (URLs S3)
  dateCreation: Timestamp
  dateModification: Timestamp
}

Order {
  id: UUID (PK)
  acheteurId: UUID (FK → User)
  vendeurId: UUID (FK → User)
  montantTotal: Decimal
  statut: Enum (EN_ATTENTE, CONFIRMEE, LIVREE)
  modePaiement: Enum (MOBILE_MONEY, CASH)
  dateCreation: Timestamp
  dateModification: Timestamp
}

OrderItem {
  id: UUID (PK)

  prixUnitaire: Decimal
  sousTotal: Decimal
}

Review {
  id: UUID (PK)
  produitId: UUID (FK → Product)
